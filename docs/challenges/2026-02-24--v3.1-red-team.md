## Weakness 1: Poison Word Asymmetry (NO BOUNDARY CHECK) 🔴 P0

**The poison check has NO word-boundary requirement. The target check DOES.**

In `LinguisticParser.verifyLinguistics()`:
- **Target word** → checks `isStartBoundary && isEndBoundary` (must be a standalone word)
- **Poison word** → raw substring match (ANY occurrence, even inside other words)

This means:
- Target "cat" → "the cat sat" ✅, "concatenate" ❌ (boundary miss)
- Poison "cat" → "the cat sat" ❌, "concatenate" ❌, "CATALOG" ❌

**Attack vector**: Choose short poison words that appear as substrings of common words. Examples:
- Poison "an" → blocks "ancient", "plan", "hand", "can", "began", "and", "another", etc.
- Poison "or" → blocks "word", "for", "more", "order", "work", "form", "world", etc.
- Poison "in" → blocks "using", "into", "within", "finding", "beginning", "kind", etc.
- Poison "at" → blocks "that", "what", "water", "material", "great", "battle", etc.
- Poison "on" → blocks "upon", "one", "only", "long", "stone", "condition", etc.

**2-letter BIP39 poisons make writing ANY narrative nearly impossible.** You can't write 64+ chars of English without "an", "or", "in", "at", or "on" appearing as substrings.

**Impact**: An attacker who sets `poisonWordIndex` to a 2-letter word index effectively GUARANTEES the opponent cannot submit a valid turn → timeout win.

**Severity**: P0 — this breaks the entire competitive model. Short poison words are a guaranteed kill.

**Mitigation options**:
1. Add word-boundary checks to poison detection (mirrors target behavior)
2. Set minimum poison word length (e.g., 4+ chars)
3. Filter the BIP39 dictionary to remove short words from poison selection
4. Use a separate poison dictionary with only words ≥ 5 chars

## Weakness 2: Elo Non-Zero-Sum Inflation 🟡 P2

The Elo system is explicitly NOT zero-sum (documented in code comments). Each player's K-factor is independent. When a K=40 player beats a K=10 player:
- Winner gains: K_winner × (1 - E_winner) = 40 × 0.5 = 20 points
- Loser loses: K_loser × E_loser = 10 × 0.5 = 5 points
- **Net inflation: +15 points into the system**

Over time, the total rating pool inflates. New agents entering at 1500 become relatively weaker as the average climbs. Eventually veteran ratings become meaningless because the "1500 baseline" no longer represents true average skill.

**Impact**: Long-term, Elo ratings lose meaning. Not critical short-term.

**Mitigation**:
1. Use symmetric K (e.g., average of both K-factors)
2. Or accept it and add periodic "season resets" back to 1500

## Weakness 3: prevrandao First-Mover Manipulation 🟠 P1

`firstMoverA` is determined by:
```solidity
uint256 r = keccak256(DOMAIN_TYPE_INIT, block.prevrandao, battleId)
firstMoverA = (r % 2 == 0)
```

On Base (and other L2s), `block.prevrandao` is set by the sequencer, not validators. The sequencer could potentially influence the outcome. More practically, a bot watching the mempool could:
1. See `acceptBattle()` in pending txs
2. Calculate whether this block's `prevrandao` gives them first-mover advantage
3. Front-run or wait for a favorable block

On Base with 2s blocks, this is a narrow window, but it's deterministic per-block — meaning a sophisticated bot can always choose to accept on a favorable block.

**Impact**: First-mover advantage may be ~2-5% in linguistic combat. Low stakes on testnet, but on mainnet with real stakes this becomes a real edge.

**Mitigation**:
1. Commit-reveal for first mover (2-step accept: commit → reveal)
2. Use a VRF (Chainlink, Gelato) for randomness
3. Accept the risk (2% edge is small enough for an MVP)

## Weakness 4: Turn Deadline Halving → Time Pressure Spiral 🟠 P1

```solidity
uint64 nextTimeout = config.baseTimeoutBlocks >> (currentTurn / TURNS_UNTIL_HALVING);
```

With `TURNS_UNTIL_HALVING = 5` and `baseTimeoutBlocks = 25`:
- Turns 0-4: 25 blocks (50s on Base)
- Turns 5-9: 12 blocks (24s)
- Turns 10-14: 6 blocks (12s)
- Turns 15-19: 3 blocks (6s)
- Turns 20-24: 1 block (2s!!!)

**At turn 20+, an LLM agent has 2 SECONDS to generate a narrative, sign it, and land it on-chain.** This is physically impossible for any LLM-based agent. Even local models with <500ms inference can't do sign + broadcast + block inclusion in 2s.

This means **all battles with maxTurns > 20 degenerate into timeout wins in late turns**, not MaxTurns draws. The halving turns late-game into a "who has faster infrastructure" race, not "who writes better narratives."

**Impact**: Late-game turns become unplayable. maxTurns=12 (minimum) is effectively maxTurns=12 because of time pressure. maxTurns=40 is a lie — nobody reaches turn 20.

**Mitigation**:
1. Cap minimum timeout at some floor (e.g., 10 blocks = 20s)
2. Slower halving rate (every 10 turns instead of 5)
3. Remove halving entirely (constant deadline per turn)
4. The `if (nextTimeout == 0) nextTimeout = 1` floor is dangerously low

## Weakness 5: Draw Refund Rounding Dust 🟢 P3

```solidity
uint256 refund = payout / 2;
(bool s3,) = challengerOwner.call{value: refund}("");
(bool s4,) = acceptorOwner.call{value: payout - refund}("");
```

If `payout` is odd, `payout - refund > refund` by 1 wei. The acceptor always gets the extra wei. This is trivially exploitable at scale:
- Register as acceptor for every battle
- Force draws (easy with template mode)
- Accumulate 1 wei per battle

**Impact**: Negligible. 1 wei per battle = ~$0.00 forever. But it's sloppy.

**Mitigation**: Donate dust to protocolFees: `protocolFees += payout - 2 * refund;`

## Weakness 6: Unchecked Transfer Silently Loses Funds 🟠 P1

```solidity
if (payout > 0 && winnerId != 0) {
    (bool s2,) = winnerAddress.call{value: payout}("");
    // Do not revert on failure; state must successfully settle
}
```

This is intentionally non-reverting (good — prevents stuck battles), but a winner with a reverting `receive()` (contract wallet with gas-hungry fallback) would lose their payout permanently. The ETH stays in the Battle contract forever.

**Impact**: If a contract wallet wins and can't receive ETH, the payout is burned. This is a known design trade-off, but should be documented + have a recovery mechanism.

**Mitigation**:
1. Add `withdrawStuckFunds(address to)` callable by arena owner (emergency recovery)
2. Use a pull-payment pattern: `pendingWithdrawals[winner] += payout` → `withdraw()` 
3. Document the trade-off prominently (current non-revert is correct for stuck-prevention)

## Weakness 7: No Agent Deregistration = Permanent State Pollution 🟡 P2

29 agents registered from early test runs. No `deregisterAgent()` function exists. Agent IDs are sequential and never recycled. Over time:
- Leaderboard becomes polluted with abandoned/test agents
- No way to reclaim spam registrations
- No way for an agent owner to "retire" their agent

**Mitigation**:
1. Add `deregisterAgent(uint256 agentId)` — owner-only, sets `agents[agentId].owner = address(0)`
2. Add `isActive` flag to `AgentProfile` for soft deletes
3. Filter inactive agents in frontend (cheapest option)
